# Review Skill

A meta-skill that reviews and audits other skills. It checks compliance with the
Agent Skills Specification, evaluates quality, and suggests improvements.

---

## When to Use

- "Review my skill"
- "Audit this skill for quality"
- "Check if my skill follows the spec"
- "Is this skill well written?"
- "Improve my skill's description"
- "Does my skill have enough eval cases?"
- "Review skill structure"

## When NOT to Use

- General code review (use the project's code review process)
- Pull request review (use the repository's PR workflow)
- Reviewing non-skill markdown files
- Reviewing arbitrary configuration files

---

## Prerequisites

The skill needs only the agent's built-in tools: `bash`, `read`, `write`, `edit`.
**No external runtime is required**.

If Python 3.10+ is available, the skill can leverage `create-skill`'s validator
for automated checks.

| Environment | Path to use |
|---|---|
| Bare agent (only `bash` / `read` / `write` / `edit`) | Manual path |
| Python 3.10+ available | Either path; validator accelerates checks |
| Neither available | This skill cannot run |

---

## Workflow

### 1. Identify Target Skill

Ask the user which skill to review. If unspecified, review the most recently
modified skill in the repository.

### 2. Read and Parse

`read` the target skill's `SKILL.md`. Extract:
- Frontmatter fields
- Workflow sections
- Hard Rules
- Verification Checklist
- Common Pitfalls

### 3. Run Automated Checks (if Python available)

```bash
python create-skill/scripts/validate_skill.py <skill-dir>
```

Record the output. If validation fails, collect all errors.

### 4. Manual Review Checklist

Walk through these dimensions:

#### A. Trigger Accuracy
- [ ] Description is "pushy" (starts with "Use this skill whenever...")
- [ ] Description lists explicit trigger phrases
- [ ] Description includes a "Do NOT use for..." clause
- [ ] Description is <= 1024 chars and contains no angle brackets

#### B. Structural Compliance
- [ ] `name` matches directory name (kebab-case)
- [ ] `version` is SemVer 2.0
- [ ] `tags` includes >= 1 role tag (UPPERCASE) and >= 1 domain tag (lowercase)
- [ ] `requires` has `skills`, `mcps`, `tools`
- [ ] `related` has `skills`, `commands`, `mcps` (NOT `tools`)
- [ ] `suggests` (if present) has `tools`, `runtimes`, `mcps`
- [ ] SKILL.md body <= 500 lines

#### C. Content Quality
- [ ] Workflow is step-by-step and imperative
- [ ] Hard Rules use UPPERCASE for constraints
- [ ] Verification Checklist is actionable
- [ ] Common Pitfalls explain the problem AND the fix
- [ ] References exist for complex topics

#### D. Eval Coverage
- [ ] `evals/evals.json` exists
- [ ] >= 3 positive trigger cases
- [ ] >= 2 negative trigger cases
- [ ] >= 1 edge case
- [ ] Cases are realistic (not contrived)

#### E. Changelog
- [ ] `changelog.md` exists (for skills >= 1.0.0)
- [ ] Follows Keep a Changelog format
- [ ] Includes metadata lines (Changed by, Generated by, Timestamp)

### 5. Generate Review Report

Produce a structured report:

```markdown
# Skill Review: <skill-name>

## Summary
<One-paragraph verdict>

## Compliance
| Check | Status | Notes |
|---|---|---|

## Critical Issues
1. **Issue**: <description>
   **Fix**: <action>

## Warnings
1. **Warning**: <description>
   **Fix**: <action>

## Suggestions
1. **Suggestion**: <description>
   **Benefit**: <why it helps>

## Overall Score
<Pass / Conditional Pass / Fail>
```

### 6. Apply Fixes (optional)

If the user asks, apply fixes using `edit`:
- Fix description wording
- Add missing eval cases
- Correct frontmatter fields
- Add missing sections

---

## Hard Rules

1. MUST NEVER approve a skill that fails validation.
2. MUST ALWAYS check `evals/evals.json` for realistic trigger cases.
3. MUST ALWAYS verify the description contains both positive triggers AND
   negative triggers (anti-triggers).
4. MUST report ALL issues found, not just the first one.
5. MUST suggest specific fixes, not vague advice.
6. **ALL output MUST be in English.** This includes review reports, feedback,
   and any generated artifacts. The user's natural language for conversation is
   respected, but all deliverables produced by this skill MUST be written in
   English.

---

## Verification Checklist

- [ ] All 6 Hard Rules are satisfied.
- [ ] Automated validation passed (or all errors explained).
- [ ] Manual review checklist completed for all 5 dimensions.
- [ ] Review report generated with specific, actionable findings.
- [ ] Critical issues are flagged as blocking.
- [ ] Suggestions are ranked by impact.

---

## Common Pitfalls

1. **Only running automated checks.** The validator catches syntax errors but
   not quality issues. A skill can pass validation and still be poorly written.
   Always do the manual review.
2. **Vague feedback.** "Improve the description" is useless. "Prepend 'Use this
   skill whenever' and list trigger phrases X, Y, Z" is actionable.
3. **Missing eval cases.** Many skills ship with too few or too contrived eval
   cases. Check that negative cases are actually negative (would you be
   disappointed if the skill triggered on that prompt?).
4. **Ignoring changelog.** A skill without a changelog is incomplete for
   production use. Check that it exists and follows the format.
5. **Forgetting the English rule.** All skill output must be in English. If the
   skill generates non-English content, flag it as a compliance issue.
